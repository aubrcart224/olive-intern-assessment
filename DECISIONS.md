## Quiz spec schema

<!-- What shape does a generated quiz take? Why those fields? What did you deliberately leave out? -->

- Versioned JSON spec with a quiz object
- Supports miltile choice, yes no, slider. No free text or images. Free text ruins the scroing as it needs to be done manually, and images would either need to be pre determined or generated by the model leaving images to be generated is unreliable.
- Normalized 0-100 scoring model so result bands always map cleanly
- Score delta on every option rather than arbitray scoring functions
- Kebeb-case IDs enforced via regex for stability.
- Adding too much just leads to a less relibale on coherent llm output. As well as a lot of ambiguity to the frontend. Keeping things simple leads to a fun and
  reliable expeince for the user.

## Question type vocabulary

<!-- Which question types does your system support? Which did you skip and why? -->

- Supports multiple choice, single or multi select, yes/no, slider.

- Skipped a lot of the more complext options such as free text images and more complex logic. You could have a option to include all of these but i think it should be selected for by the user before the prompt rather than up to the LLM for what to include. Also ruins the logic in regards to scoring as some things might need manual scoring such as the text box, which could be useful depending on what you were looking for. For the sake of the project i focused on having a purely llm driven quiz that always functions reliably and is fun to take with consitaint scoring logic.

## Scoring & results logic

<!-- How does scoring work? Weighting? Branching? Multiple result dimensions? What's in vs. out of scope? -->

- Model: normalized_100. Each answer contributes a scoreDelta. The frontend computes theoretical min/max possible score across all questions, then linearly maps the user's raw score to 0-100.
- Result bands: Percentage ranges (e.g., 0-25, 26-50). The schema enforces contiguous 0-100 coverage.
- Branching can override: A branch can force a specific result band (destination.kind === "result") or skip to end.
- What's out: No weighted questions, no multi-dimensional results (e.g., "You scored 80% on health, 40% on budget"), no custom scoring formulas. Kept it linear and predictable.

## Edit loop

<!-- After the first generation, how does a user iterate? Full regeneration, spec patching, or direct editing? Why? -->

- The user gets to see all of the questions and can either edit them manually or re generate them individually. This allows the user to see what theyre quiz looks like before they deploy it. Allowing the user to change things independanly ensures they dont have a random luck of the draw on their quizes and are able to keep what they like and change what they dont or simply edit the wording of a question. Users can also edit how much each question is worth if they feel the ai has wrongly assigned a value.

## Prompt reliability

<!-- How do you validate LLM output? Retries? Fallbacks? What happens when it returns garbage? -->

- Two-phase strategy: Initial generation → Zod validation. If it fails, send a repair prompt with the exact validation issues and raw output back to the LLM for one self-correction attempt.
- Extraction: extractJsonCandidate() strips markdown fences and finds the first {...} block in case the model adds prose.
- Parsing: Zod schema with superRefine for semantic checks (duplicate IDs, unknown branch destinations, result band gaps).
- Fallback: If repair also fails, the API returns 422 with the raw output and issues list so the frontend can show the user what went wrong. No silent degradation.
- Some cost tradeoff, one extra LLM call only when needed (~N% of the time).

## Cost

<!-- Approximate $ per generated quiz. Show your math. -->

• Primary generation: ~1 Claude Sonnet 4.5 call at 4000 tokens max. Typical input ~1.5k tokens (system prompt + user prompt), output  
 ~1.5-2.5k tokens.
• At ~$3/1M input tokens and ~$15/1M output tokens: roughly $0.005–$0.02 per quiz.
• Repair call: Only on validation failure. Same model, smaller output usually. Adds ~$0.003–$0.01 when triggered.
• Option regeneration: ~2000 tokens max. Much cheaper, ~$0.002–$0.005.
• Ballpark: ~$0.01–$0.03 per generated quiz, cheaper if it passes validation on the first try.
